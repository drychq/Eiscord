\section{总体描述}

\subsection{产品定位}
类 Discord 实时社区交流系统是一套面向兴趣社区、课程小组和团队协作场景的在线交流平台。系统以“社区组织 + 频道协作 + 实时互动”为核心，通过统一账号体系、灵活的社区结构和即时通信能力，为用户提供持续在线的交流环境。系统默认以 Web 端为行为基线，同时支持桌面端与移动端共享核心业务逻辑。

\subsection{版本范围与优先级}
本 SRS 将需求划分为 P0、P1 和 P2 三个优先级。P0 为 MVP 必须实现范围，应进入第一轮开发、联调和验收；P1 为增强能力，可在 P0 稳定后迭代；P2 为明确暂不实现或仅保留扩展空间的能力。

\noindent\begin{tabularx}{\textwidth}{L{2cm}L{3.2cm}Y}
  \toprule
  优先级 & 开发含义 & 范围说明 \\
  \midrule
  P0 & MVP 必选 & 注册登录、个人资料、好友申请、私聊、社区创建与加入、频道管理、文本消息、附件元数据、提及、未读、通知、角色权限、语音频道成员状态同步。 \\
  P1 & 后续增强 & 密码找回、多端细节适配、离线推送、社区搜索增强、审计查询界面、更多消息管理能力。 \\
  P2 & 暂不实现 & 真实多人语音媒体流、屏幕共享、机器人平台、开放接口市场、商业化订阅、广告投放、复杂内容审核和独立运营后台。 \\

\subsection{MVP 成功标准}
当系统满足以下条件时，可认为 v1 MVP 具备开发交付价值：
\begin{itemize}
  \item 用户能够完成注册登录、维护基础资料、添加好友并进行一对一私聊。
  \item 用户能够创建或加入社区，在具备权限的文本频道中发送消息、附件元数据和提及消息。
  \item 社区所有者或管理员能够创建频道、维护角色、分配权限，并阻止无权访问或无权操作。
  \item 通知、未读和在线状态能随关键事件更新，文本消息端到端可见时间满足非功能指标。
  \item 语音频道在 v1 中仅要求成员加入、退出、静音、闭麦和断线状态同步，不要求承载真实音频流。
\end{itemize}

\subsection{典型使用场景}
\begin{enumerate}
  \item 新用户完成注册后，通过邀请码加入某个课程讨论社区，浏览文本频道并与成员交流。
  \item 两名已互加好友的用户通过私聊进行一对一沟通，系统实时同步双方在线状态与未读提醒。
  \item 某兴趣社区创建多个文本频道和语音频道，将不同主题的讨论分区管理，便于成员协同。
  \item 社区管理员为不同成员授予角色，限制某些频道的访问权限，保证信息隔离与管理秩序。
  \item 社区成员在消息中提及其他用户，系统向被提及用户推送实时通知并更新未读状态。
\end{enumerate}

\subsection{设计约束}
\begin{itemize}
  \item 本系统属于课程设计项目，需求表达应聚焦核心功能，不扩展为开放生态平台。
  \item 文本消息、在线状态、通知和语音成员状态默认采用事件驱动的实时同步方式，本文档仅定义逻辑接口和事件契约，不绑定具体框架或数据库。
  \item 多端产品共享统一账号体系和业务规则，但本文档不细化原生客户端差异化特性。
  \item 社区管理能力仅覆盖与即时交流直接相关的成员、频道和权限控制，不包含独立运营后台。
  \item 所有涉及访问控制的功能均以最小权限原则为前提，权限不足的用户不得执行越权操作。
\end{itemize}

\subsection{关键假设}
\begin{itemize}
  \item 用户具备稳定的网络连接，并使用支持现代 Web 标准的客户端环境。
  \item 平台已具备基础邮件、短信或等效身份验证通道，以支持注册验证和密码找回。
  \item 语音频道在 v1 中采用状态同步 MVP，不在本版本内提供真实语音媒体传输、直播、屏幕共享或机器人伴随功能。
  \item 单个社区规模上限为 5000 名成员，系统整体需支撑 10000 名注册用户和 1000 名并发在线用户。
  \item 文件上传主要面向消息附件、头像与社区图标，不涉及在线协作文档编辑或复杂媒体处理流程。
\end{itemize}
